Converting Inline Styles to CSS Classes

One of the first things I learned to use AI for was converting messy inline SVG styles into organized CSS classes. This was tedious work when I did it manually, but with the right prompt, AI handles it perfectly. The key is being specific about what you want. üéØ

üü† Before AI Help

‚ùå Messy inline styles with mixed color formats:

<path fill="#ffefcb" d="..."/>
<path fill="rgb(253,207,116)" d="..."/>
Prompt I Used:

"Refactor all inline fill and stroke attributes into CSS classes. Use hsl() colors and CSS custom properties in :root. No inline styles."

‚úÖ AI Response

Clean CSS with hsl() colors and custom properties:

<style>
  :root {
    --bulb-glow: hsl(45 100% 89%);
    --bulb-warm: hsl(40 97% 73%);
  }
  .bulb-glow { fill: var(--bulb-glow); }
  .bulb-warm { fill: var(--bulb-warm); }
</style>

<g class="bulb-glow">
  <path d="..."/>
</g>
<g class="bulb-warm">
  <path d="..."/>
</g>

HAP's Note: AI followed my rules perfectly once I explained them. My job was to rename the groups semantically and verify that the visual result stayed identical.

üéì HAP's Rules for Working with AI

1

Be Specific

Tell AI exactly what to do and why. "Clean this SVG" is too vague. "Use hsl() colors and CSS custom properties" is clear.

2

Add Format Requirements

Always say "Use hsl() for colors" and "No inline styles" in your prompts. AI won't know your standards unless you state them.

3

Use CSS Custom Properties

Never allow hard-coded color values. Require AI to create custom properties in :root for all colors.

4

No Inline Styles

Everything belongs in CSS classes or :root. Inline styles break theming and make maintenance harder.

5

Ask for an Implementation Plan

Read the plan first and spot flaws before execution. This catches problems before they're coded.

6

Stay the Reviewer

AI assists you ‚Äî you approve the final code. Never trust output without testing it yourself. üü†

Advanced Prompt Strategies

Once I got comfortable with basic prompts, Prof. Teeters taught me that real AI workflow involves iteration and refinement, not one-shot perfection. These advanced prompts help me improve AI output, troubleshoot problems, and prevent common mistakes. üî¨

Iteration Prompt (Refinement):

"Revise this output. Keep all hsl() colors and CSS classes, but rename generic class names like .group-1 to semantic names that describe their visual purpose."

HAP's Use Case: AI's first pass often works but uses generic names. This prompt helps me add meaning without redoing the whole conversion.

Troubleshooting Prompt (Visual Validation):

"Compare this optimized SVG to the original and identify any visual differences. List what changed and explain why those changes occurred."

HAP's Use Case: When optimization goes wrong, this prompt helps me understand what AI removed and why. Then I can give better instructions.

Safety Prompt (Preserve Critical Elements):

"Refactor this SVG without altering any ID attributes, class names, or aria-labelledby references. Preserve all accessibility markup exactly as written."

HAP's Use Case: I learned this one the hard way! When my SVG has JavaScript hooks or ARIA connections, this prompt prevents AI from breaking them.

HAP's Workflow Tip:

I rarely use just one prompt anymore. My typical workflow: (1) Basic prompt to get structure right, (2) Iteration prompt to improve naming, (3) Troubleshooting prompt if something looks wrong, (4) Safety prompt before final optimization. Prof. Teeters calls this "conversational refinement" ‚Äî treating AI like a colleague I'm collaborating with, not a magic wand I wave once.

When NOT to Use AI
